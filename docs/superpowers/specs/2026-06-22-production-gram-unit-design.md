# 生产录入克单位换算 — 设计文档

**日期**：2026-06-22
**分支**：newtool
**目标**：让生产录入时可用更精确的单位（克），但出入库依旧使用现有库存单位（袋/罐/瓶等），降低人工操作成本。

---

## 背景与问题

当前系统中：

- 物品库存单位为袋/罐/瓶/包/条/盒/件/克等
- 配方用量 `product_bom.qty_per_unit` 精度为 2 位小数
- 配方示例：焦糖海盐每件需 0.48 罐焦糖酱 + 0.02 瓶海盐

问题：像酱料、海盐、牛乳这类**按重量称取**的原料，配方里用「0.48 罐」这种小数表达，工人无法直接称重，必须心算「0.48 罐 ≈ 多少克」，既慢又易错。需要让工人**直接照着克数称重**。

约束：

- 出入库、库存查阅、回退等流程**仍使用现有库存单位**，不引入双套单位记账
- 降低人工操作成本是核心目标（方案 C「仅提高精度」被否，因为工人仍按罐的小数思考）

---

## 核心架构判断

**克是「录入和显示」单位；库存单位（袋/罐/瓶）仍是「库存记账」单位。**

换算只发生在两个边界：

1. **配方录入时** —— 启用克的原料用克录入
2. **生产提交时** —— 克数自动换算回库存单位去扣减

出库、回退、删除、库存页面**全部保持库存单位不变**。这样「出入库依旧使用现有单位」自然成立，回退逻辑（`quantity + actual_qty`）完全不用改。

---

## 设计

### 1. 数据模型

唯一的 schema 改动 —— `items` 表加一列：

```sql
ALTER TABLE items ADD COLUMN gram_per_unit REAL NOT NULL DEFAULT 0;
```

- `gram_per_unit = 0` → 该物品**不启用**克（默认，所有现有物品保持原状）
- `gram_per_unit > 0` → 该物品启用克，值表示「1 库存单位 = 多少克」（如牛乳填 `1000` 表示 1 袋 = 1000 克）

`product_bom.qty_per_unit` 的语义按物品动态决定（**不加新列**）：

- 物品启用克 → 这里存**克数**（如 480）
- 物品没启用克 → 这里存**库存单位量**（如 0.48 罐），和现在完全一样

**迁移方式**：与现有 `migrate_warehouse_db_columns()` 一致 —— 在 `db/__init__.py` 的 `WAREHOUSE_SCHEMA` 加列定义，并在迁移函数里加一个 `PRAGMA table_info(items)` 检查，缺列就 `ALTER`。每个仓库 db 在请求时自动补列，无需手动操作。

### 2. 换算逻辑

新增纯函数（放 `_helpers.py`），整个特性的数学核心：

```python
def grams_to_stock(grams: float, gram_per_unit: float) -> float:
    """克 → 库存单位。gram_per_unit=0 时原样返回（未启用克）。"""
    if gram_per_unit <= 0:
        return grams
    return float((Decimal(str(grams)) / Decimal(str(gram_per_unit)))
                 .quantize(Decimal('0.01')))
```

**生产提交时的换算流程**（`production.submit`）：

| 步骤 | 启用克的物品（牛乳，1袋=1000克） | 未启用的物品（甜筒，按条） |
|------|-------------------------|----------------------|
| 配方 `qty_per_unit` | 480（克/件） | 2（条/件） |
| 产出 3 件，本批需 | 480 × 3 = **1440 克** | 2 × 3 = **6 条** |
| 折算扣库存 | 1440 ÷ 1000 = **1.44 袋** | **6 条**（不变） |
| 写入 `actual_qty` / 出库 / 扣减 | **1.44 袋**（库存单位） | **6 条** |

关键点：

- 换算只在「克 → 库存单位」这一个方向，发生在提交那一刻
- 写进 `production_run_items.actual_qty`、`outbound_requests`、`stock_movements` 的全是**库存单位**（1.44 袋）
- 因此**回退、删除、出库、库存页面零改动** —— 它们看到的永远是库存单位
- 库存不足校验（`actual > stock`）也用换算后的库存单位比较，逻辑不变

**精度处理**：换算除法用 `Decimal` 量化 2 位小数（已确认「扣小数袋」）。本批总需克数**先算总和再除**，避免逐行累积误差。

### 3. 配方录入界面（product_edit.html + 物品编辑页）

**物品编辑页**加一个可选字段：

```
每单位克重 [____] 克   （留空或 0 = 不启用克，按原单位录配方）
```

放在物品的单位字段旁边。如牛乳填 1000。

**配方编辑页** 的 BOM 行 —— 录入单位**跟随所选原料动态切换**：

- 选中的原料启用了克 → 该行用量输入框标签显示「**克**」，提示「每件产品需要的克数」
- 选中的原料没启用克 → 标签显示原单位（罐/瓶/条），和现在一样

实现上，`_load_items_for_bom()` 的查询已经 `SELECT i.unit`，只需多带一个 `i.gram_per_unit`。前端用一小段 JS：选择原料下拉变化时，根据该物品是否有 `gram_per_unit` 切换该行的单位标签。

**录入示例**（焦糖海盐配方）：

| 原料 | 启用克? | 录入 | 含义 |
|------|--------|------|------|
| 原味奶浆 | 否 | 1 袋 | 每件 1 袋 |
| 焦糖酱 | 是(1罐=2000克) | 960 克 | 每件 960 克 |
| 海盐 | 是(1瓶=500克) | 10 克 | 每件 10 克 |

存进 `qty_per_unit` 的就是 `1`、`960`、`10`，语义由各自物品的 `gram_per_unit` 决定。

### 4. 生产录入界面（session.html）

工人每天操作的页面，目标是**让工人直接照着克数称重**，不用心算。

**配方卡**（左侧静态）—— 每单位用量，启用克的显示克：

```
原味奶浆      1 袋
焦糖酱      960 克
海盐         10 克
```

**原料卡**（右侧动态）—— 产出量变化时实时计算「本批用量」。假设产出 3 件：

| 原料 | 库存 | 本批用量 | 工人操作 |
|------|------|---------|---------|
| 原味奶浆 | 50 袋 | **3 袋** | 取 3 袋 |
| 焦糖酱 | 5.2 罐 | **2880 克** | 称 2880 克 |
| 海盐 | 8 瓶 | **30 克** | 称 30 克 |

关键设计：

- 启用克的原料，「本批用量」直接显示**克数**（2880 克），工人照着称重，无需换算
- 库存栏仍显示**库存单位**（5.2 罐），因为出入库用库存单位 —— 工人能对照「还剩几罐」
- 库存不足判断：把本批克数换算成库存单位后和库存比（2880÷2000=1.44 罐 < 5.2 罐 ✓），红色预警逻辑不变
- 提交时，hidden field 传**库存单位量**（已换算），后端拿到就是 1.44 罐，复用现有扣减路径

**JS 改动**：现有 `recompute()` 函数已按 `qty_per_unit × output_qty` 算本批量。只需给每行多带 `data-gram-per-unit`，启用克的行：

- 显示克数 = `qty_per_unit × output_qty`
- hidden `actual` 值 = `克数 ÷ gram_per_unit`（库存单位）

两个数分离：**显示给人看克，传给后端的是库存单位**。

**历史记录页**（runs.html）：`actual_qty` 存的是库存单位，照常显示「1.44 罐」即可，无需改动。可选增强（启用克的原料额外括注克数）暂不做（YAGNI）。

## 测试方案

按要求：**测试用独立 agent 执行**，且**必须有实际操作验证**（不只单元测试）。

### 第一层：单元测试（pytest，`grams_to_stock` 纯函数）

- `grams_to_stock(1440, 1000)` → 1.44
- `grams_to_stock(6, 0)` → 6（未启用克，原样返回）
- `grams_to_stock(2880, 2000)` → 1.44
- 边界：0 克、极小量（10÷500=0.02）、量化精度（1000÷3 → 333.33）

### 第二层：集成测试（pytest + Flask test client，跨边界验证）

- 建启用克的物品 + 配方 → 提交生产 → 断言库存按**库存单位**正确扣减
- 断言 `outbound_requests` / `stock_movements` 写入的是库存单位
- **回退测试**：生产后回退，库存精确还原（验证零改动路径仍正确）
- 混合配方：一个产品同时含克原料和非克原料，各扣各的

### 第三层：实际操作验证（独立 agent + preview 工具，真实点击）

- 启动应用，agent 登录
- UI 创建启用克的物品（牛乳，1袋=1000克）
- UI 创建配方（焦糖海盐，牛乳 480 克/件）
- 进生产页，选产出量 3 件 → **截图验证页面显示「1440 克」给工人看**
- 提交 → **验证库存从 X 袋变成 X-1.44 袋**
- 回退 → 验证库存还原
- 全程截图留证

### 执行分工

- **代码由 haiku 模型实施**（writing-plans 阶段会把任务拆成 haiku 可执行的小步）
- **测试由独立 agent 执行** —— 第三层实操验证派一个独立 agent，用 preview 工具真实操作，与写代码的上下文隔离

### 成功标准

三层全绿 + 实操截图证明工人看到的是克、库存扣的是库存单位 + 回退精确还原。

## 已确认的设计决策

1. 核心流程：配方填克 → 提交时自动换算回库存单位扣减 ✓
2. 适用范围：混合模式，按物品决定（启用克的用克，没启用的用原单位）✓
3. 换算率存储：物品表加 `gram_per_unit` 字段 ✓
4. 库存扣减：扣小数库存单位（1.44 袋），保留 2 位小数 ✓
5. 实现方案：方案 A（克作录入/显示，库存单位作记账）✓
6. 五段设计（数据模型/换算逻辑/配方界面/生产界面/测试方案）逐段确认 ✓
